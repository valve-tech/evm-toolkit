# `@valve-tech/evm-toolkit` — API docs

Per-package API documentation, regenerated by `yarn docs:build` and
gated in CI by `yarn docs:check`. **All files in this directory are
generated** — do not hand-edit. To update, edit the source TSDoc
comments in `packages/<name>/src/`, then re-run `yarn docs:build` and
commit the result.

## What's here

| File / dir | Format | Purpose |
| --- | --- | --- |
| `<package>.json` | TypeDoc JSON model | Machine-readable artifact for downstream docs ingestion |
| `<package>/index.html` | TypeDoc HTML | Self-contained browsable docs (open directly in a browser) |
| `trueblocks-openapi.yaml` | OpenAPI 3.x | Vendored copy of the upstream chifra daemon's REST API spec, pinned to the same SHA `scripts/codegen.mjs` uses |
| `manifest.json` | Custom (see below) | Index file listing every package's artifacts; the single entry point a docs-site consumer reads to discover what's available |

## `manifest.json` schema (v1)

```jsonc
{
  "schemaVersion": 1,
  "generatedFor": "docs.valve.city ingestion",
  "repository": "valve-tech/evm-toolkit",
  "packages": [
    {
      "name": "@valve-tech/chain-source",
      "slug": "chain-source",
      "version": "0.17.0",
      "description": "...",
      "html": "chain-source/index.html",
      "json": "chain-source.json"
    },
    {
      "name": "@valve-tech/trueblocks-sdk",
      // ...as above, plus:
      "openapi": "trueblocks-openapi.yaml"   // only on this package
    }
  ]
}
```

When a `package` entry contains an `openapi` field, the file at that
relative path is the OpenAPI spec for the HTTP API the package wraps —
render it alongside the TS surface (most downstream sites use Redoc
or Swagger UI for OpenAPI, and `typedoc-plugin-markdown` or a custom
renderer for TypeDoc JSON).

## `<package>.json` shape (TypeDoc model)

These are emitted by [TypeDoc](https://typedoc.org) with
`--json`. The schema is TypeDoc's own — see
[the TypeDoc JSON output guide](https://typedoc.org/guides/json/). At
the top level:

- `name` — `@valve-tech/<name>`
- `packageName`, `packageVersion`
- `children[]` — array of every exported symbol, each with `name`,
  `kind` (1=Project, 2=Module, 64=Function, 128=Class, 256=Interface,
  …), `comment.summary[]`, `comment.blockTags[]`, `signatures[]`,
  `type`, `flags`
- `symbolIdMap` — flat `name → id` lookup

The richest field for human-readable docs is `comment.summary[]` (the
JSDoc body) and `comment.blockTags[]` (parsed `@param`, `@returns`,
`@example`, `@link` references).

Common rendering pipeline downstream:

1. Read `manifest.json` to discover packages.
2. For each `package.json` artifact, fetch the file and walk
   `children[]`.
3. For each child with `kind === 64` (function), `256` (interface),
   `128` (class), or `2097152` (type alias), render a section using
   `comment.summary` as the description body and `signatures[]` as
   the signature display.
4. Resolve `{@link OtherSymbol}` references by looking up `name` in
   the same package's `symbolIdMap`, or in another package's manifest
   entry for cross-package links.

## Why per-package output instead of one monolithic file

Each package is independently versioned and published to npm under
its own name. Consumers may render the docs for one package at a time
(e.g. on a per-package landing route), so per-package artifacts are
the more useful unit. The `manifest.json` ties them together.

## How `docs.valve.city` ingests this

The docs site pulls files from `raw.githubusercontent.com/valve-tech/
evm-toolkit/main/docs/api/` (or pins to a release tag for stable
links). The recommended flow:

1. Fetch `manifest.json` first.
2. For each package, fetch `<package>.json` and render the TypeDoc
   model.
3. For `trueblocks-sdk`, additionally fetch `trueblocks-openapi.yaml`
   and render it under the same package landing page (one route
   surfaces both the TS surface and the upstream HTTP spec the SDK
   wraps).

## CI gate

`yarn docs:check` re-runs the build into a temp location and diffs
the JSON output against committed state. If they don't match, the
gate fails. To recover: run `yarn docs:build` locally and commit the
regenerated files.

The check exists for the same reason
`scripts/persisted-types.manifest.json` exists: any change that
modifies the surface (new export, renamed field, edited TSDoc block)
must be acknowledged by committing the regenerated artifact, not
silently shipped.
